# Tab `2.2 General Skills` ‚Äî Analysis

Source JSON: `.tmp/sheets/2.2_General_Skills.json`
Sheet name: `2.2 General Skills`
Dimensions: 132 rows x 16 cols (max_row=132, max_col=16), 1304 non‚Äëempty cells, 26 merged ranges (all in column A ‚Äî they merge consecutive rows that share a category label).

---

## 1. Purpose

This tab is the leaf‚Äëlevel skills ledger. Each row is one general skill (108 named skills + ~14 blank custom/specialization slots = ~122 row slots). For every skill the player allocates ranks (`# Ranks`); the sheet computes the Final Total skill value as:

```
Total = Ranks Value + Cat Value + Stat Value + Talent + Special
```

Cat Value is inherited from tab `2.1 Categories` (the category total for the skill's parent category). Stat Value is the bonus from tab `1. Stats & Traits` for the skill's governing characteristic. Costs (DP per rank) are *hardcoded per row* in column F based on the governing stat's cost class (in=2, pr=4, others=3).

The header at `O1` (`='Level Progression'!J3` = 178 in this character) shows the character's available DP pool from level progression ‚Äî `P1` label "Available DPs to allocate" is just a sticky note next to it. There is no spent‚ÄëDP rollup formula on this tab; rank cost rolls up elsewhere (probably `Level Progression` or `Playing Dashboard`).

## 3. Per‚Äërow formula pattern

Take row 4 (`Animals ‚Üí Hunting`, stat `in`):

| Cell | Formula | Value |
|------|---------|-------|
| A4   | (blank ‚Äî A is merged from A2) | ‚Äî |
| B4   | `=if(A4="",B3,A4)` | `Animals` (category resolver) |
| C4   | (literal) | `Hunting` |
| D4   | (literal) | `in` |
| F4   | (literal) | `2` |
| G4   | (user input) | `1` |
| H4   | `=if(or(G4=0,G4=""), 0, MIN(if(G4<=10, G4*3, 30) + if(G4<=10, 0, if(G4<=20,(G4-10)*2,20)) + if(G4<=20,0,(G4-20))))` | `3` |
| I4   | (literal) | `10` (category total mirrored from `2.1 Categories!K8`) |
| J4   | (literal) | `12` (stat total mirrored from `1. Stats & Traits!G19` for `in`) |
| K4   | (user input) | `0` |
| L4   | (user input, often empty) | (empty) |
| M4   | `=H4+I4+J4+K4+L4` | `25` |
| O4   | `='2.1 Categories'!O10` | `null` (stale ref ‚Äî column O of categories is unused there) |

**Inputs (user‚Äëeditable):** A (only first row of category), C (skill name, optional for custom slots), D (stat code), F (cost; auto by stat), G (ranks), K (talent), L (special).
**Computed:** B (category resolver), H (ranks value), M (total).
**Mirrored from other tabs (currently stored as constants):** I (category total), J (stat total).

### Ranks Value (column H) ‚Äî piecewise progression

```
H = if (G is 0 or blank) 0
    else MIN(
        (G <= 10 ? G*3 : 30)        // first 10 ranks: +3 each (cap +30)
      + (G <= 10 ? 0 : (G <= 20 ? (G-10)*2 : 20))  // ranks 11‚Äì20: +2 each (cap +20)
      + (G <= 20 ? 0 : (G - 20))    // ranks 21+: +1 each (no cap inside MIN)
    )
```

Equivalent table:
- 0 ranks ‚Üí 0
- 1 ‚Üí 3, 2 ‚Üí 6, ... 10 ‚Üí 30
- 11 ‚Üí 32, 12 ‚Üí 34, ... 20 ‚Üí 50
- 21 ‚Üí 51, 22 ‚Üí 52, ...

The outer `MIN(...)` has only one argument, so it's effectively a no‚Äëop ‚Äî likely vestigial. Mirrors `rank_bonus_progression` in DB but this is **per‚Äëskill (3/2/1)**; the categories tab uses **(5/2/1)** in `G2` (the Trait/Category formula) and **(3/1/0.5)** in `G8+`. The progression to mirror in the frontend for skills is `3 / 2 / 1`.

### Total (column M)

```
M = H + I + J + K + L
  = ranks_value + cat_value + stat_value + talent + special
```

Notice `L` (Special) is referenced by every Total formula even though no row in the captured data has a numeric L value (all empty ‚Üí treated as 0).

## 4. Recurring patterns

1. **Category resolver in B:** `B_n = if(A_n="", B_(n-1), A_n)` ‚Äî fills every row with its category label so `B` can be used as a clean lookup key.
2. **F (Cost) stat-keyed:** in‚Üí2, pr‚Üí4, all other stats‚Üí3, custom blank rows‚Üí`-`.
3. **H (Ranks Value)** ‚Äî same piecewise formula every row, only the row index changes.
4. **M (Total)** ‚Äî same `H+I+J+K+L` shape every row.
5. **I (Cat Value)** ‚Äî *constant* per skill but identical for all skills sharing a category. Should be a formula referencing `2.1 Categories` but is currently denormalized.
6. **J (Stat Value)** ‚Äî *constant* per skill, equals `1. Stats & Traits!G_<row of stat>`. Same denormalization.
7. **O column** ‚Äî leftover formula `='2.1 Categories'!O<n>`; the target cells are empty so all O values evaluate to `null`. Probably dead‚Äëcode / abandoned.

## 7. DP cost rules

- **Per‚Äërank cost** is a flat constant per skill, *governed by the skill's stat*, not by the character's stat total:

  | Stat (col D) | Cost (col F) |
  |---|---|
  | in | 2 |
  | pr | 4 |
  | st, co, ag, re, sd | 3 |

  Confirmed by inspecting all stat/cost combinations in the sheet (no exceptions). Empty / custom slots show `-` (no cost until the user names the skill).

- **DP spent on a skill = `ranks (G) * cost (F)`.** The sheet has a header "DP ALLOCATION" over column E but the per‚Äërow E cells are **empty** ‚Äî the breakdown is never written back. The total spent must be aggregated upstream (likely on `Level Progression`).

- **DP cap** is `O1 = 'Level Progression'!J3 = 178` for this character.

- This per-skill cost differs from `stat_cost_rules` (DB), which is keyed on *stat total*. The sheet's constants suggest the convention "in skills cost 2 DP/rank because Insight is cheap; Presence skills cost 4 DP/rank". The DB schema's `stat_cost_rules.dp_cost` row is keyed on the stat-total bucket and is shared with categories/weapon-skills/traits ‚Äî the comment "applies to categories, skills, weapon skills and traits using their governing stat" implies that here it would be **the cost of buying ranks in the category, mirrored to the skill via the stat that governs the category**. Either the sheet author hardcoded the 2/3/4 values from a `stat_cost_rules` lookup (keyed by stat code, not stat total), or the rule is "cost = stat_cost_rules.dp_cost for stat row in 1.Stats!D column". Verify against `1. Stats & Traits!D13..D19` ‚Äî those D values were `2,3,9,6,5,4,10` (stat values) and `D13..D19` of stats sheet showed cost-class numbers? See open question below.

## 8. Specializations

- The sheet exposes specialization slots in two ways:
  1. **Named placeholder rows** with text containing `(   )`, e.g. `Species (                  )` (rows 56‚Äì58, Flora & Fauna), `Branch*` (row 62, Magic Lore). The user types the specialization inline.
  2. **Empty C cells** within a category block (rows 37, 38, 42, 43, 48, 49, 82, 83, 94, 95, 99, 100, 105, 106, 120, 121). These are *generic* free slots ‚Äî the user can write any custom skill name within that category.
- There is **no DM‚Äëapproval workflow visible in this tab**. The `character_skill_access` table (specialization_name + approval_status) is a frontend addition; the sheet treats specializations as plain rows with user-typed names.
- Mapping: rows in (1) ‚Üí `skills.allows_specialization=true` and an entry in `character_skill_access` with `specialization_name` filled. Rows in (2) ‚Üí `skills.is_custom=true` (a per-character custom skill in the same category).

## 9. Custom skills

Yes ‚Äî many rows have `is_custom`-style affordance:
- 16 blank C rows across 9 categories (Crafts Fine, Crafts Gross, Crafting Lores, Performing, Science Analytic, Socio Economics, Social Sciences, Technical/Trained ‚Äî 2 slots each plus 1 for Power Sensibility-adjacent uses).
- These rows still execute the H/M formulas, so the moment the user types a name and ranks, they auto‚Äëcompute.
- Category, stat, and cost are pre‚Äëfilled even when C is blank, so the DB representation is "a `character_skills` row with a custom `skills` entry whose `is_custom=true` and `category_id` matches".

## 10. Open questions

1. **Where is `dp_spent` aggregated?** The "DP ALLOCATION" header at E1 implies a per‚Äërow breakdown was planned but never filled. Confirm by inspecting `Level Progression!J3`'s formula tree ‚Äî does it subtract `SUMPRODUCT(2.2!F:F, 2.2!G:G)`?
2. **Why is column I (Cat Value) a constant rather than `='2.1 Categories'!L<n>`?** Likely the sheet author hand‚Äëcopied to keep the layout shape; the frontend should re‚Äëderive it. Verify the sheet stays in sync after editing categories ‚Äî it probably doesn't.
3. **Talent (K) source.** Does `K` ever read from `Talents tables` / `TALENTS|FLAWS LIST`? In the captured data K is always the literal `0`. Need a populated character to confirm. The DB has `talent_bonus` so the frontend should let users type per‚Äëskill talent grants and sum them.
4. **Specialization placeholder syntax.** Are `Species (    )` / `Branch*` / `Spell: ‚Ä¶` the *only* sanctioned specialization patterns, or is any open‚Äëparen a marker? (DB allows arbitrary `specialization_name`.)
5. **Spell-bound skills (rows 85, 86)** ‚Äî only two slots are wired (`'2.4 Magic'!A2`, `A3`). What if the character knows >2 spells? This is a sheet limitation; the frontend should auto-generate one Power Sensibility row per known spell.
6. **The dead `O2..O130` references** ‚Äî confirm they can be deleted, or whether they were meant to surface category metadata that the categories tab no longer exposes in column O.
7. **Cost rule ‚Äî DB vs sheet.** DB `stat_cost_rules` is keyed on stat *total*, but the sheet's per‚Äërow F is keyed on stat *code* (constant per stat). Decide which authoritative source the frontend uses; if `stat_cost_rules` is keyed on stat total, the formula `dp_cost = stat_cost_rules[character.stats[D].total]` may yield 2/3/4 *because* Insight has a high stat total in this character ‚Äî but for a different character it could change. **Confirm with `1. Stats & Traits!D13..D19` (the D column on the stats tab is labeled "Bonus" or "DP cost"?).**
8. **Why are rows 7‚Äë10 "Athletic Nimble" only 4 skills (Acrobatics, Contortions, Jumping, Wrestling) but the DB likely lists 5?** Possibly a missing row ‚Äî or this is exactly the named‚Äëskill list. Cross‚Äëcheck with `skills` table contents.
9. **`qu` (Quickness) is not used anywhere in column D.** Confirm no general skill uses Quickness as governing stat ‚Äî only weapon skills and resistance categories do.
10. **Professional / Activity / Manual / Temp modifiers** are absent from the sheet but present in the DB. Confirm whether any are computed elsewhere (e.g. `Combat Dashboard`, `Playing Dashboard`) and feed back to the skill total at runtime.
