# 2.1 Categories

## Purpose

This tab is the **per-character DP allocation grid** for the three Anima-style skill-grouping families ("Categories"):

1. **Traits** (rows 2–7) — six special "trait categories" that aren't normal skills: Endurance, Power Points, General Knowledge, RR Will, RR Stamina, RR Magic. These have a different DP-cost-per-rank profile (the `*5 / *2 / +1` curve) and pull their `Cost` and `Stat Value` from the `1. Stats & Traits` tab directly.
2. **General** categories (rows 8–33) — the 26 "skill bundles" governed by a single primary stat (Animals, Athletic Nimble, Awareness, Combat Styles, Crafts, Outdoors, Subterfuge, Urban, Warfare, etc.).
3. **Weapons** categories (rows 34–42) — the 9 weapon-family categories (Chains, Crush & Cleave, Swords, Misile, Range Modern, Polearms, Thrown, Thrust, Slings).

The merged cells `A43:A48` (max_row=49) imply a fourth section (likely "Mystic / Psychic" or a Combat / supernatural family group) that exists in the template but is empty for this character.

For each category the player allocates DP into `# Ranks` (col F). The sheet then computes a `Ranks Value` bonus from a piecewise progression, sums it with the governing-stat bonus, talent bonus, and special bonuses to produce the category's `Total` and a final `Modified` value (Total + a global "Playing Dashboard" all-skills modifier).

The total DP spent here flows back to the dashboard via N1 = `'Level Progression'!J3` (DP available at this level).

## Per-row formula breakdown

### Traits (rows 2–7)

For row R in [2,7] with a Trait at row R-23 in `1. Stats & Traits` (R=2 → row 25, R=3 → row 26, …, R=7 → row 30):

- **C (Stat label):**
  `=CONCATENATE('1. Stats & Traits'!B<25+R-2>," / ",'1. Stats & Traits'!C<25+R-2>)`
  Builds the "co / sd" label for the two-stat pair (primary / secondary) the trait is governed by.
- **E (Cost):**
  `='1. Stats & Traits'!D<25+R-2>`
  DP cost per rank, looked up from the Stats & Traits tab.
- **F (# Ranks):** editable input.
- **G (Ranks Value):**
  ```
  =if(or(F<R>=0,F<R>="")
     , 0
     , MIN(
         if(F<R><=10, F<R>*5, 50)
       + if(F<R><=10, 0, if(F<R><=20, (F<R>-10)*2, 20))
       + if(F<R><=20, 0, rounddown((F<R>-20),0))
       )
   )
  ```
  This is the **Trait progression curve**:
  - Ranks 1–10: +5 per rank (max +50)
  - Ranks 11–20: +2 per rank (max +20 more)
  - Ranks 21+: +1 per rank
  - 0 ranks ⇒ 0 (no penalty for traits, unlike skills)
  - The outer `MIN(...)` is a noop with one argument — likely vestigial.
- **H (Stat Value):** `='1. Stats & Traits'!F<25+R-2>` — the trait's "stat value" / current bonus from the stats tab.
- **I (Talent):** manual entry (rows 2–7 have hand-entered talent contributions: Endurance 45, PP 0, GK 20, Will 22, Stam 15, Magic 18).
- **J (Special):** manual entry; empty here.
- **K (Total):** `=G<R>+H<R>+I<R>+J<R>`
- **L (Modified):**
  - Row 2 (Endurance): `=K2` (no global skill modifier applies to Endurance)
  - Rows 3–7: `=K<R>+'Playing Dashboard'!$B$7`

### General categories (rows 8–33) and Weapons categories (rows 34–42)

Rows 8 through 42 share an identical structure (the only thing that changes is the category name in B, the stat code in C, and the DP cost in E):

- **B:** static name string.
- **C:** static 2-letter stat code (`ag` / `co` / `in` / `pr` / `re` / `sd` / `st`). Not a formula — frontend will resolve to `categories.stat_id` via name → code → `stats` lookup.
- **E:** DP-cost-per-rank, hard-coded 2 / 3 / 4 (depends on the category's primary stat — most stats cost 3, intelligence-based cost 2, presence-based cost 4).
- **F:** editable input.
- **G (Ranks Value):**
  ```
  =if(or(F<R>=0,F<R>="")
     , -25
     , MIN(
         if(F<R><=10, F<R>*3, 30)
       + if(F<R><=10, 0, if(F<R><=20, (F<R>-10), 10))
       + if(F<R><=20, 0, rounddown((F<R>-20)/2,0))
       )
   )
  ```
  This is the **Skill progression curve**:
  - 0 ranks ⇒ **−25** (the "untrained penalty")
  - Ranks 1–10: +3 per rank (max +30)
  - Ranks 11–20: +1 per rank (max +10 more)
  - Ranks 21+: +1 per 2 ranks (rounded down)
  - Outer `MIN(...)` again is vestigial / single-argument.
- **H (Stat Value):** appears as a hard-coded number (7, 12, 4, etc.) but the value is the governing stat's `bonus` from `stat_progression`. In a Next.js port this becomes a **derived display value** — never edited here. The current spreadsheet snapshots the value rather than referencing it.
- **I (Talent):** manual; all 0 in this character.
- **J (Special):** manual; all 0 in this character.
- **K (Total):** `=G<R>+H<R>+I<R>+J<R>`
- **L (Modified):** `=K<R>+'Playing Dashboard'!$B$7`

## Recurring patterns

- The **Trait pattern** repeats over rows **2–7** (6 rows).
- The **Skill (General + Weapons) pattern** repeats over rows **8–42** (35 rows).
- Rows **43–48** are reserved by merged range `A43:A48` but contain no cells in this snapshot — likely a fourth group (Mystic? Combat Style? Psychic?) that this character hasn't unlocked.
- **Total category rows in template: 41 visible (6 traits + 26 general + 9 weapons), with up to ~6 reserved.** The DB `categories` table has 35 rows, which matches General(26) + Weapons(9). Traits are stored separately (likely as `traits` table) and the merged-but-empty A43:A48 group might be a planned addition.

## DP cost rules

There are **two layered concepts**, and both apply per-row:

### 1. DP cost *per rank* (the `Cost` column, E)

This is **looked up** from the governing stat's "skill cost" entry — i.e. the `stat_cost_rules` / `stat_progression.cost_skill` table, indexed by the stat's *value* (not bonus).

For Traits (rows 2–7), the formula `='1. Stats & Traits'!D25` pulls a precomputed value from the Stats tab. For General/Weapons, the value is hard-coded (3 = strength/agility/dex/constitution-based, 2 = intelligence/perception-based, 4 = power/presence-based) — so in the spreadsheet these are static, but logically they correspond to the same `stat_progression.cost_skill` lookup.

**Rebuild rule:** for a category governed by stat S with current value V,
```
cost_per_rank = stat_progression.cost_skill[V]
```
(or equivalently `stat_cost_rules` keyed by stat_total). The current spreadsheet just denormalizes this.

### 2. Rank → bonus translation (the `Ranks Value` column, G)

This is the **`rank_bonus_progression`** table. Anima uses two different curves depending on the category type:

- **Skill curve** (General + Weapons, rows 8–42):

  | Ranks | Bonus |
  |---|---|
  | 0 | −25 |
  | 1 | +3 |
  | n ≤ 10 | +3·n |
  | 10 | +30 |
  | 11 ≤ n ≤ 20 | 30 + (n−10) |
  | 20 | +40 |
  | n > 20 | 40 + floor((n−20)/2) |

- **Trait curve** (rows 2–7):

  | Ranks | Bonus |
  |---|---|
  | 0 | 0 |
  | n ≤ 10 | 5·n |
  | 10 | +50 |
  | 11 ≤ n ≤ 20 | 50 + 2·(n−10) |
  | 20 | +70 |
  | n > 20 | 70 + (n−20) |

These two curves should be stored as `rank_bonus_progression` rows with a discriminator for `category_group` ∈ {`trait`, `skill`}.

### Total DP spent (not stored explicitly on this sheet)

Despite the header "DP ALLOCATION" and the cell N1 = `'Level Progression'!J3` (total DP available), the sheet does **not** sum `Σ(F·E)` or `Σ(ranks·cost)` to a "total DP spent" cell on this tab. That aggregation likely lives on the `Level Progression` tab (where N1 sources from) or on `Playing Dashboard`. Recommend adding a `Σ ranks·cost` cell in the rebuild UI for visual feedback.

## Open questions

1. **What sits in rows 43–48?** Merged label `A43:A48` exists but no cells. Likely a 4th group (Mystic? Psychic? Combat-style?). Cross-check the master template (a fresh / level-1 character sheet) or other character snapshots.
2. **Is the global `'Playing Dashboard'!$B$7` per-character or per-session?** The label there will tell us if this is a long-term modifier (e.g. armor's "natural penalty"), a temporary status, or both. It is applied uniformly to every category except Endurance — that asymmetry needs explanation.
3. **Why is `Endurance` (L2) the only row without `+ 'Playing Dashboard'!$B$7`?** Probably because Endurance contributes to HP rather than acting as a skill check, so a global skill modifier shouldn't apply. Confirm with the rules.
4. **Talent bonus source** — rows 2–7 have non-trivial talent values (45/0/20/22/15/18) but rows 8–42 are all 0. Are talent contributions only relevant to Traits in this system, or has this character not bought any general-skill talents yet?
5. **Are Traits really "categories"?** They're laid out together but use a different curve, different stat representation (two stats), different default (0 instead of −25), and a different global-modifier rule. The DB should likely model them as a sibling table (`character_traits`) rather than reusing `character_categories`.
6. **Cost column for Traits** is `3, 3, 3, 3, 3, 2` — pulled from `Stats & Traits!D25:D30`. Is this static per character (set during creation) or recomputed when stats change? The rebuild should probably recompute.
7. **`# Ranks` (col F) is the input, but the DB column is `dp_allocated`.** Decide canonical storage. Storing ranks is more user-friendly; storing DP makes invariants (DP-budget) trivial. Recommend: store **ranks** and compute spent-DP as a derived value.
8. **Outer `MIN(...)` wrapper** in both G formulas takes only one argument, so it's a no-op. Likely vestigial from a draft that capped the bonus. Safe to drop in the port.
