# 2.5 Talents&Flaws

## Purpose

This tab is the **player-facing summary view** for the character's selected Talents and Flaws ‚Äî NOT the full catalog (that lives in `TALENTS|FLAWS LIST` and `Talents tables`). It is laid out as a static three-section list:

1. **Racial Talents** (rows 2‚Äì9, left side) ‚Äî talents the character gains because of their race; not paid for with DP.
2. **Main Talents** (rows 10‚Äì17, left side) ‚Äî discretionary "main" talents bought with DP.
3. **Secondary Talents** (rows 18+, left side) ‚Äî discretionary "secondary" talents bought with DP.
4. **Flaws** (rows 1+, right side, columns F‚ÄìH) ‚Äî flaws the character has taken; cost is **negative** (refunds DP).

A single live formula in `J1` shows the **DP budget remaining** to spend on talents/flaws, pulled from the master Level Progression sheet.

The tab is largely a **manual list/form** ‚Äî the player types names, costs, descriptions directly into the rows. There are no `VLOOKUP`/`INDEX-MATCH` formulas wiring rows to the catalog tab in this snapshot. Only one formula exists (`J1`).

## Cross-tab references

Only **one** formula in this tab:

```
J1 : ='Level Progression'!J3
```

This pulls the **Available DP pool** (178 in this snapshot) from the Level Progression sheet's cell `J3`. That value is the budget the player must stay within when summing the costs in column B (Main+Secondary) and column G (Flaws ‚Äî negative, so they extend the budget).

There are **no** `VLOOKUP`/`INDEX-MATCH` references to `TALENTS|FLAWS LIST` or `Talents tables` in this snapshot. Names, costs, and descriptions are typed by the player. (In the full sheet there is presumably a manual data-entry workflow; the catalog tabs serve as a reference, not a live lookup source for this view.)

## Total cost / DP accounting

- `J1` shows the **available** DP budget from Level Progression.
- This tab does **not** compute a running sum of talent costs vs. budget in any visible cell. It is the player's responsibility to keep the sum of costs ‚â§ J1.
- Flaws use **negative** costs (e.g. `-15`), which conceptually add to the budget ‚Äî the player can buy more talents in exchange for taking flaws.
- The expected accounting (implicit, to be implemented in the Next.js UI):
  - `talents_spent = SUM(Main_Talents.Cost) + SUM(Secondary_Talents.Cost) + SUM(Flaws.Cost)`
  - `remaining = J1 - talents_spent`
  - Racial Talents are free (no cost column for them).

## Choice handling

The sheet does **not** capture per-talent runtime choices (e.g. "+2 to a stat of choice", "pick a weapon", "pick a category"). The free-text Description column is the only place such choices are recorded ‚Äî the player is expected to write the chosen option into the description. There is no structured slot for `stat_id`, `category_id`, `skill_id`, or `weapon_id`.

This is exactly what the DB table `character_talent_choices` is for in the Next.js port ‚Äî it formalizes what the spreadsheet leaves implicit.

## How a player adds a talent (UX flow inferred from sheet)

In the spreadsheet (the source of truth for player intent):

1. Player consults `TALENTS|FLAWS LIST` / `Talents tables` to pick a talent.
2. Player types `Name`, `Cost`, `Req`, `Desc` into the next free row of Main or Secondary Talents (or Flaws on the right).
3. Player mentally tracks `SUM(costs) ‚â§ J1` (= 178 DP from Level Progression).
4. Racial talents are pre-listed by GM/race rules ‚Äî player typically doesn't edit those.
5. Choice-bearing talents: player writes the chosen stat/skill/weapon into the Desc cell as free text.

In the Next.js + Supabase port (recommended replacement flow):

1. **Display top bar**: `Available DP: 178 | Spent: X | Remaining: 178 - X` (live from `character_talents` joined to `talents.cost` √ó `times_taken`, plus negative flaw costs).
2. **Three sections** (Racial / Main / Secondary) + **Flaws** section, each rendering rows from `character_talents` joined to `talents` filtered by `talent_type`.
3. **Add Talent button** opens a picker:
   - Filter by `talent_type` for the section.
   - Filter out talents the character already has at `times_taken >= max_times_per_character`.
   - Filter out `is_level_1_only = true` if character is past level 1.
   - Show requirements (`talent_requirements`) and validate against character stats/categories/skills before allowing add.
   - Flag `requires_dm_approval` and require a DM-approval toggle.
4. **On select**, if the talent has any `talent_*_bonuses` rows marked as "choice" (or has multiple eligible targets), present a **choice modal** that writes to `character_talent_choices` (one row per choice).
5. **Validation**: block submit if `total_cost > J1 (available DP)`. Flaws lower `total_cost` because their `talents.cost` is negative.
6. **Racial talents** are auto-populated from the chosen race (separate flow, not a player picker).
7. **Remove**: deletes the `character_talents` row + cascade-deletes its `character_talent_choices`.

The Next.js UI replaces the free-text spreadsheet layout with a structured catalog-driven picker, and replaces the implicit "write your choice in the description" pattern with structured `character_talent_choices` rows.
