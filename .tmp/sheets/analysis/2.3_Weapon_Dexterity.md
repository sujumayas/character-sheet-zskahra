# Tab `2.3 Weapon Dexterity` — Analysis

Source JSON: `/Users/esenespinosa/Development/personal/projects/Zskahra - Character Sheet/.tmp/sheets/2.3_Weapon_Dexterity.json`
Sheet metadata: `name="2.3 Weapon Dexterity"`, `max_row=25`, `max_col=61`, 1305 non-empty cells, 9 merged ranges.

## 1. Purpose

Tracks per-weapon mastery (the "Weapon Dexterity" / Attack table in Anima: Beyond Fantasy). Each weapon has its own skill total. Two mechanics make this tab interesting:

1. **Per-weapon DP investment** — the player buys ranks in a specific weapon at a per-weapon DP cost; that produces a "Ranks Value" using the same step-function used by every other skill (3/rank up to 10, 2/rank 11–20, 1/rank thereafter).
2. **Weapon-affinity transfer-in** — partial mastery in any other weapon flows into this weapon via a fixed penalty table. The displayed Total for a weapon is the **maximum** across "directly trained" and "best transferred-in value from any other trained weapon".

Note: this tab does **not** compute the full N+talent+special pipeline that the DB schema implies. Many fields the DB stores (talent_bonus, professional_bonus, temp/manual/activity, natural_value, adolescent/package ranks) are absent or collapsed into single hand-entered cells here. The sheet is a stripped-down version compared to `character_weapon_skill`.

## 4. Affinity matrix (the heart of the tab)

### Block R–AM (cols 18–39): static **affinity penalty** table

Pure data, no formulas. Each row has 22 numeric values aligned with the 22 column headers. Most pairs are 0 (no transfer relationship). Non-zero entries are negative penalties:
- The **diagonal** (self vs self) is 0 (the row's own column).
- Sibling weapons within the same family receive a small penalty (e.g. row 4 = 1h Maces: `U4=-30` to 1h Axes, `W4=-30` Short Swords, `X4=-30` Long Swords, `AS=Short Swords` etc.).
- More distant relationships get larger penalties (`-50`, `-63` etc.) up to "no transfer at all" (left as 0 → blank in derived matrix).
- Convention: a non-zero R–AM cell **enables** transfer (with the penalty); a 0 means "no affinity, do not transfer". This is why the AN–BI formula gates on `R=0 → ""`.

This block matches the DB's `weapon_affinity` table (88 rows of source→target with `affinity_modifier`). Non-zero R–AM cell at `(target_weapon, source_weapon_col)` ↔ DB row `(source_weapon_id=col_weapon, target_weapon_id=row_weapon, affinity_modifier=value)`. There are far fewer than 22×22=484 entries because 0-cells are "no row", which matches the DB count of 88.

Quoted example values (Row 4 = 1h Maces target):
- `R4=0, S4=0, T4=0, U4=-30, V4=-50, W4=-30, X4=-30, Y4=-50, Z4=0, AA4=0, AB4=0, AC4=0, AD4=-50, AE4=-50, AF4=-50, AG4=0, AH4=0, AI4=0, AJ4=-50, AK4=-50, AL4=0, AM4=0`

### Block AN–BI (cols 40–61): derived **transfer-in skill value** table

Every cell is a formula of identical shape (484 cells in total — 22 rows × 22 cols). Verbatim from `AN2`:

```
=if($Q2=AN$1, $N2,
    if(R2=0, "",
        VALUE(INDEX($C$2:$N$23, match(AN$1,$C$2:$C$23,0), 12)) + R2))
```

Reading the formula:
- `$Q2 = AN$1`? — i.e. is this the diagonal (the weapon transferring to itself)? If yes, return its own Natural value `$N2`.
- Else, if `R2 = 0` (no affinity entry for this pair) → blank.
- Else, look up the source weapon's **`L` column value** in `$C$2:$N$23` (the 12th column of that range, counting `C` as 1 → `N`, but actually `C..N` is 12 columns and column 12 is `N` = "Natural VALUE"). Then add the affinity penalty `R2`.

So `AN–BI[row r, col c]` = `( source weapon c ).N (Natural value) + ( target row r , source col c ).R..AM (penalty)` — i.e. "what is row `r`'s effective skill if it borrows training from column `c`".

The shown Total `M_r = max(AN_r..BI_r)` thus picks the best option across (own training) and (every viable transfer-in). The pull-source weapon's column header `AN$1..BI$1` is matched against `C$2:$C$23` (skill names) by `MATCH`, so the matrix is order-independent in principle (works as long as the column headers exist as skill names).

Subtle bug to watch in the port: `INDEX($C$2:$N$23, ..., 12)` returns column index 12 inside C..N which is column `N` → `Natural VALUE`. Re-implementing in code, use `weapon_skill[source].natural_value` plus `weapon_affinity[source→target].affinity_modifier`.

## 5. Recurring patterns

- **Rank-cost step function** in column H is the same pattern used in `2.2 General Skills` (3·n / 2·n / 1·n breakpoints at 10 / 20). DB equivalent: `rank_bonus_progression`.
- **Cat / Stat / Talent / Special** stack: `Total = ranks_value + cat + stat + talent + special` (column N). Identical to the general-skills pipeline minus professional_bonus / temp_modifier / etc.
- **Category-fallthrough**: column B uses `=if(A="",B↑,A)` so merged category cells don't break per-row logic.
- **Diagonal-vs-transfer max**: the `M = max(AN:BI)` idiom encodes "use the best of trained + transferred" in one cell. This is the only place this pattern appears in the sheets seen so far.
- **Static table + formula derivation**: clean separation between the affinity data block (R–AM) and computed derivation (AN–BI). Mirrors the DB design (data in `weapon_affinity`, derivation in app code).

## 6. Cross-tab references

Only one cross-tab formula in the entire tab:
- `O1`: `='Level Progression'!J3` — the player's current total DP. Displayed in the header but never used in any cell formula on this tab (no DP cap enforcement here).

Notable **missing** cross-tab links (all baked as hand-entered values on this sheet):
- `D` (Stat code) is not used; `J` (Stat Value) is not pulled from `1. Stats & Traits`. In the port, `J` should be derived from the linked stat's bonus.
- `I` (Cat Value) is not pulled from `2.1 Categories`. In the port, this should come from the character's combat archetype / category attack bonus.
- `K` (Talent) is not pulled from `Talents tables` or `2.5 Talents/Flaws`.
- No link to `Adolescence ranks` (free weapon ranks) or `Training Packages` (package weapon ranks). The DB tracks `adolescent_ranks` and `package_ranks` separately, but the sheet only has one `# Ranks` cell — the user must fold those into G manually.

## 8. DP cost

Per weapon: `cost_dp = F * G` = `weapons.dp_cost * character_weapon_skill.ranks`.

The sheet does **not** sum these up or compare against `O1` (Level Progression DP). DP enforcement is presumably aggregated in `Level Progression` via cross-tab sums. In the port, compute and display:
- `Σ (cost_dp)` across all weapons (a single tab-level totals row),
- compare against the character's available DP budget (per `stat_cost_rules` / `Level Progression`).

## 9. Open questions

1. **Static cat/stat/special values** — are these intentionally frozen (so the player customizes per-weapon at sheet build time), or did the original author just bake them and forget to wire formulas? The port should make them live joins; we may want a "manual override" toggle to preserve the spreadsheet author's freedom.
2. **`MIN(...)` with one arg in H formula** — is harmless but suggests the author originally wanted `MIN(cap, computed)`. Likely a leftover; safe to drop in the port.
3. **`L` column dual purpose** — has both negative (situational penalties like -15 Short/Long Swords) and positive (e.g. +40 on 2H Swords) values. Map to `special_bonus` (signed) or split into `special_bonus` + `special_modifier` per DB columns. Recommend: rename DB `special_bonus` semantics to "any signed special".
4. **Specialization** — DB has `weapons.allows_specialization` and `character_weapon_skill.special_modifier`; not represented on this tab. Likely surfaces as a toggle in the port.
5. **Folding adolescent / package / talent ranks into G** — the sheet asks the player to do mental math. The port should expose `ranks` (paid), `adolescent_ranks`, `package_ranks` as separate inputs and compute `total_ranks` for the H step function.
6. **No DP cap or budget validation** on this tab — confirm whether `Level Progression` sums DP from this tab or whether the sheet is "trust the user".
7. **Diagonal M behavior** — `M = max(AN:BI)` always includes the diagonal (own `N` value), so `M ≥ N` always. Confirm in the port that we never display `M < N`.
8. **22 weapons × 2 weapon_types** — only 2 types in DB, so most are "melee" with a few "ranged". Verify mapping of bows/crossbows/pistols/rifles/projectiles/sling/slingshot.
9. **Negative totals are valid** (e.g. `M2 = -13`) — they represent "cannot effectively use this weapon"; in the port, allow signed display, no clamping.
